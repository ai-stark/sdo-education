Standards are intended to enable products to more easily communicate with each other and share information, even when products are developed by different companies. Outside of health IT, standards are often used to allow products and devices to work together. For example, you probably have a number of devices that use Bluetooth technology to connect together, like your smartphone, a fitness tracker, printers, or even your car. These devices are all able to connect to each other because they use a set of standardized Bluetooth profiles that tell the devices how to connect. Another example is ATMs. When ATMs were first introduced, you could only use the ATM at the bank that issued your card. Over time, the manufacturers of ATMs agreed to a set of standards that would allow you to use any ATM in the world, regardless of which bank issued your card.

There are a range of standards that cover everything from how data is coded to how it is shared, including but not limited to:

Vocabulary/Terminology standards: Vocabulary standards are all about how individual data elements are structured, recorded, and transmitted. For example, when a provider adds a problem to a patient’s chart in your EHR system, the system will automatically link that problem to either an ICD-10 code or a SNOMED-CT code. Likewise, when a provider orders a medication for a patient, the system will automatically link the medication to an RxNorm code that will be electronically transmitted to the pharmacy, so that they fill the prescription with the right medication. Note that vendors have done this by mapping codes on the back-end of the system.

Format standards: Rarely is a single clinical data element for a patient shared electronically with another system, meaning, that typically a summary of clinical information (including medications, problems, allergies, lab results, etc.) is shared, rather than just one element like medication. When a health IT system receives this summary information, it has to be able to interpret it, which requires knowing what data will be in what locations, and what vocabulary the data will use. This is done via format standards, like the C-CDA (a standard summary of care document used for transitions of care) that layout for a developer where data is at in the document and the vocabulary for each data element.

Transport standards: Transport standards are how the data gets from point A to point B. We use transport standards every day in the non-clinical world when we send emails. Prior to the adoption of SMTP standards, individuals with a Yahoo email address could not share emails with someone with a Hotmail address. Transport standards allow for the secure exchange of data across the internet, but each product has to follow the same transport standard or exchange will not occur. While some transport standards, like Direct are very basic and similar to email, other transport standards allow for more complex data connectivity to share a broader set of data, or APIs, like, FHIR that will allow users to query for a single data element.

While standards make it significantly easier to exchange data across health IT systems, they can’t fix all of the issues with exchanging data. Standards are intended to support a broad number of use cases, often across multiple countries. While standards are intended to address specific business needs, they are also intended to meet the broadest set of needs with a single standard. So sometimes standards allow variation that can make it harder to exchange data for a particular use case or within a particular country whose health system operates differently than other nations. In order to narrow down the use case for a particular standard, SDOs will create an implementation guide or specification that helps guide health IT developers in how they implement the standard for a small set of use cases. Again, because multiple use cases are met with a single guide/specification, there is still allowed variation. For example, the guide/specification may suggest that a developer use a vocabulary for a particular data type, like a procedure, but it may not require that they use a vocabulary code because they want to allow for optionality to support a broader number of use cases. This type of variation can sometimes make it hard to exchange data, despite the use of standards.

Additionally, standards merely “standardize” the data that has been put into the system. There’s a common saying, “garbage in, garbage out.” So, if data quality is poor, i.e. data is not being recorded, it’s not being coded to the right vocabulary, etc., standards won’t do much to help with the exchange of the data, or help receiving systems be able to interpret the data. Finally, there can be disagreement amongst clinicians on how data should be mapped to the vocabularies, this is considered semantic variation. For example, ICD-10 has 10 codes for endocrine, nutritional and metabolic diseases. Some organizations may use all 10 codes, while others only use a few. And if the EHR system wasn’t using vocabularies before, organizations may make different decisions on how to map those codes, potentially causing issues if another organization uses a different set of codes.
